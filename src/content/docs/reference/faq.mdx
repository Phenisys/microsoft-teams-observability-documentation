## Common FAQ

### How do I get a trial license?

Use the trial request form in <a href={`${import.meta.env.BASE_URL}getting-started/license/`}>License</a> and provide your Microsoft 365 tenant ID so the license can be generated for your tenant.

### How can I validate license sizing?

Use <a href={`${import.meta.env.BASE_URL}reference/license-estimation/`}>License Estimation</a>, based on Microsoft Teams activity reports and `MSTeams_CollectionHealth` validation.

### How can I verify license file integrity?

Do not edit the license JSON content. The `signature` field validates `license_data`, and modifications invalidate the license (see <a href={`${import.meta.env.BASE_URL}getting-started/license/`}>License</a>).

### Where can the collector run?

For host-based deployment, the collector can run on Linux (recommended) or Windows with required network access and resources (see <a href={`${import.meta.env.BASE_URL}getting-started/prerequisites/`}>Prerequisites</a>).

### Which outbound network targets are required?

At minimum:

- `graph.microsoft.com`
- `login.microsoftonline.com`
- your backend endpoint (Dynatrace or Splunk)

For some Dynatrace setups, `ip-api.com` can also be required for geolocation enrichment (see <a href={`${import.meta.env.BASE_URL}collector/troubleshooting/`}>Collector Troubleshooting</a> and <a href={`${import.meta.env.BASE_URL}backends/dynatrace/configuration/`}>Configure Dynatrace</a>).

### Which ports should be opened?

The docs define required outbound hosts and HTTPS connectivity, but do not publish a single fixed port matrix for every environment. Validate against your network policy and backend endpoint requirements.

### What are the minimum CPU and memory requirements?

No fixed global baseline is documented. Guidance is to size CPU and memory for your call volume (see <a href={`${import.meta.env.BASE_URL}getting-started/prerequisites/`}>Prerequisites</a>).

### Can I enable Auto Attendant and Call Queue collection?

Yes. The collector supports:

- `ms_teams_autoattendant_collection`
- `ms_teams_callqueue_collection`

See <a href={`${import.meta.env.BASE_URL}collector/v1/configuration/`}>Configuration</a>.

### Are extra Microsoft permissions required for those features?

Yes. The permissions reference states VAAC-specific permissions are required when these features are enabled (see <a href={`${import.meta.env.BASE_URL}reference/azure-permissions/`}>Azure Permissions Reference</a>).

### What should I verify first when data is missing?

Start with:

1. auth and consent (Microsoft + backend credentials)
2. outbound connectivity
3. at least one enabled output
4. collector logs
5. backend checks (Grail/OpenPipeline or Splunk HEC/index)

See <a href={`${import.meta.env.BASE_URL}collector/troubleshooting/`}>Collector Troubleshooting</a>, <a href={`${import.meta.env.BASE_URL}backends/dynatrace/troubleshooting/`}>Dynatrace Troubleshooting</a>, and <a href={`${import.meta.env.BASE_URL}backends/splunk/troubleshooting/`}>Splunk Troubleshooting</a>.

### What is a safe first-run sequence?

Run one cycle with `--ignore_state`, verify ingestion, then enable service mode and monitor status/logs (see <a href={`${import.meta.env.BASE_URL}collector/v1/installation/`}>Installation</a> and <a href={`${import.meta.env.BASE_URL}collector/v1/service/`}>Service Management</a>).

## Dynatrace Application FAQ

### Where is application configuration managed?

In the app's **Installation & Configuration** screen. Panels and behaviors are documented in <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/configuration/`}>Application Configuration</a>.

### Do I need to redeploy after a license change?

No full redeploy is documented. The expected flow is save configuration and reload the app when prompted (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/installation/`}>Dynatrace Installation</a>).

### Can a missing/invalid license break configuration saves?

Yes. Missing or invalid license state can prevent tenant initialization and cause save failures (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/installation/`}>Dynatrace Installation</a> and <a href={`${import.meta.env.BASE_URL}backends/dynatrace/troubleshooting/`}>Dynatrace Troubleshooting</a>).

### Can configuration be validated before/after save?

Yes. Several panels include **Test**/**Validate** actions and automatic validation states (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/configuration/`}>Application Configuration</a>).

### Is `sites.csv` mandatory?

No. It is optional in collector configuration, but required for complete site mapping and to avoid unmapped-call behavior (see <a href={`${import.meta.env.BASE_URL}collector/v1/configuration/#4-optional-sitescsv-mapping`}>Configuration</a> and <a href={`${import.meta.env.BASE_URL}backends/dynatrace/dashboards/`}>Dynatrace Dashboards</a>).

### Why do I see "No Mapped Calls"?

Subnets are not mapped to site definitions. Configure and load `sites.csv` (see <a href={`${import.meta.env.BASE_URL}collector/v1/configuration/#4-optional-sitescsv-mapping`}>Configuration</a> and <a href={`${import.meta.env.BASE_URL}backends/dynatrace/dashboards/`}>Dynatrace Dashboards</a>).

### What does the "Undetermined" site status mean?

In Sites NPA status, `Undetermined` means there is not enough data to classify the site (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/sites/`}>Application Sites</a>).

### What happens if domains are not configured for Sites filtering?

The Sites page warns that calls will not be domain-filtered until domains are set (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/sites/`}>Application Sites</a> and <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/configuration/`}>Application Configuration</a>).

### Can I search calls by caller/callee identity or call ID?

Yes. Calls supports advanced filtering including participant and call ID fields (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/calls/`}>Application Calls</a>).

### What is an "empty call"?

In Calls, empty calls are entries with duration `0` and no recording data. They are included with the **Show Empty Calls** toggle (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/calls/`}>Application Calls</a>).

### Can the app show peak simultaneous calls?

Yes. The dashboards guide lists peak simultaneous calls as a typical Home/Overview widget (with possible variation by app version) (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/dashboards/`}>Dynatrace Dashboards</a>).

### What session/device details are available in call analysis?

Call Overview documents participant, stream, network, and endpoint fields (device, platform, CPU, latency, jitter, packet loss, subnet, relay/reflexive IP, protocol) (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/call-overview/`}>Application Call Overview</a>).

### Can I download raw call details?

Yes. Call Overview supports Graph call-details JSON download when Microsoft configuration is set (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/call-overview/`}>Application Call Overview</a> and <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/configuration/`}>Application Configuration</a>).

### Where is data stored?

The documented architecture stores logs in a dedicated Grail bucket (`ms_teams`) processed by OpenPipeline (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/configuration/`}>Configure Dynatrace</a> and <a href={`${import.meta.env.BASE_URL}backends/dynatrace/`}>Dynatrace Overview</a>).

### How is app access controlled?

The app includes a dedicated **Permissions** panel for role/group access, and some features require explicit Dynatrace permissions (for example site file storage permissions) (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/app/configuration/`}>Application Configuration</a>).

## Dynatrace Extension FAQ

### Does the extension run on ActiveGate?

Yes. Extension mode requires Dynatrace with ActiveGate and runs on ActiveGate, not on the former standalone host (see <a href={`${import.meta.env.BASE_URL}collector/extension/`}>Collector Extension Overview</a> and <a href={`${import.meta.env.BASE_URL}collector/extension/installation/`}>Extension Installation</a>).

### Which outputs are supported in extension mode?

Dynatrace only. Splunk and Console outputs are not available in extension mode (see <a href={`${import.meta.env.BASE_URL}collector/extension/`}>Collector Extension Overview</a>).

### How is the license configured in extension mode?

Paste the full license JSON as a string in the extension License field (not a file path) (see <a href={`${import.meta.env.BASE_URL}getting-started/license/`}>License</a> and <a href={`${import.meta.env.BASE_URL}collector/extension/installation/`}>Extension Installation</a>).

### Can the extension use Dynatrace Credential Vault?

Yes. Extension configuration supports vault-based credentials (see <a href={`${import.meta.env.BASE_URL}collector/extension/installation/`}>Extension Installation</a>).

### Where are extension logs located?

Troubleshooting paths are documented in the extension installation guide under ActiveGate log paths (see <a href={`${import.meta.env.BASE_URL}collector/extension/installation/`}>Extension Installation</a>).

## Standalone Collector FAQ

### Does standalone require ActiveGate?

No. Standalone can run on any suitable host with required access and resources (see <a href={`${import.meta.env.BASE_URL}getting-started/prerequisites/`}>Prerequisites</a>).

### Where is the license configured in standalone mode?

Set `collection_config.license_file` in `config.yaml`, pointing to the local license file path (see <a href={`${import.meta.env.BASE_URL}getting-started/license/`}>License</a> and <a href={`${import.meta.env.BASE_URL}collector/v1/configuration/`}>Configuration</a>).

### Which outputs are supported in standalone mode?

Dynatrace, Splunk, and Console outputs are available (see <a href={`${import.meta.env.BASE_URL}collector/v1/configuration/`}>Configuration</a>).

### What should I do after changing standalone configuration?

In service mode, restart the service and verify status/logs (see <a href={`${import.meta.env.BASE_URL}collector/v1/service/`}>Service Management</a>).

### How should tokens and sensitive files be protected?

Use restricted file permissions, keep tokens out of version control, and store secrets in controlled paths (see <a href={`${import.meta.env.BASE_URL}backends/dynatrace/collector-connection/`}>Collector Connection</a> and <a href={`${import.meta.env.BASE_URL}collector/v1/service/`}>Service Management</a>).
