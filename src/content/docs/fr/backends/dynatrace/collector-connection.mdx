## Configuration de l'output du Collector

Ajoutez une entrée `dynatrace` dans la section `output` de votre `config.yaml` :

```yaml
output:
  dynatrace:
    enabled: true
    dynatrace_tenant_id: "votre-tenant-id-dynatrace"
    dynatrace_api_token: "votre-token-logs-ingest"
```

### Paramètres

| Paramètre | Requis | Description |
|---|---|---|
| `enabled` | Oui | Définir à `true` pour activer l'output Dynatrace |
| `dynatrace_tenant_id` | Oui | Identifiant du tenant Dynatrace SaaS (ex. `abc12345`) |
| `dynatrace_api_token` | Oui | Token API avec le scope `logs.ingest` |

<Aside type="caution">
Protégez votre token API — utilisez des permissions de fichier restrictives et n'incluez jamais le token dans un système de contrôle de version. Stockez le fichier de configuration à un chemin accessible uniquement par le compte de service du collector.
</Aside>

## Optionnel : format des timestamps

Vous pouvez configurer le format des timestamps des enregistrements de logs envoyés à Dynatrace :

```yaml
output:
  dynatrace:
    enabled: true
    dynatrace_tenant_id: "votre-tenant-id"
    dynatrace_api_token: "votre-token"
    time_fields_format: "iso8601"
```

Valeurs supportées : `epoch_ms` (par défaut), `epoch_s`, `epoch_ns`, `iso8601`, `rfc3339`.

## Tester la connexion

Après avoir modifié la configuration, validez et testez :

```bash
ms-teams-agent validate --config ./config.yaml
ms-teams-agent test-connection --config ./config.yaml
```

`test-connection` vérifie à la fois l'authentification Microsoft Graph et la connectivité vers l'endpoint Dynatrace.

## Vérifier l'arrivée des données

Après un premier cycle de collecte réussi :

1. Dans Dynatrace, ouvrez **Logs** et filtrez par `log.source = "MSTeams_CallRecords_CallMetadata"`.
2. Confirmez que des enregistrements arrivent dans le bucket `ms_teams`.
3. Ouvrez l'application MS Teams Observability — la page Vue d'ensemble doit afficher des données live.

En cas de problème, consultez le [Dépannage](/fr/backends/dynatrace/troubleshooting/).
