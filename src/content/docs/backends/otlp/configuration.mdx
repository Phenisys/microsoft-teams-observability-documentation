## Collector Output Configuration

Add an `otel` entry under the `output` section of your `config.yaml`:

```yaml
output:
  otel:
    - name: "my-otel-backend"
      enabled: true
      endpoint: "https://your-otel-endpoint/v1/logs"
      headers:
        Authorization: "Bearer your-token"
```

### Parameters

| Parameter | Required | Description |
|---|---|---|
| `name` | Yes | Identifier for this OTLP output (used in logs) |
| `enabled` | Yes | Set to `true` to activate the output |
| `endpoint` | Yes | Full URL of the OTLP logs endpoint (must end in `/v1/logs`) |
| `headers` | No | HTTP headers (typically used for authentication) |

### Multiple OTLP Outputs

You can define multiple OTLP destinations in the same configuration:

```yaml
output:
  otel:
    - name: "grafana"
      enabled: true
      endpoint: "https://logs-prod-eu-west-0.grafana.net/loki/api/v1/push"
      headers:
        Authorization: "Basic <base64-credentials>"
    - name: "custom-collector"
      enabled: true
      endpoint: "https://internal-otel.example.com/v1/logs"
```

## TLS Configuration

By default, TLS verification is enabled. If your endpoint uses a self-signed certificate or an internal CA, you may need to configure TLS trust appropriately on the collector host.

## Validate the Connection

After editing the configuration, run:

```bash
ms-teams-agent validate --config ./config.yaml
ms-teams-agent test-connection --config ./config.yaml
```

`test-connection` verifies both Microsoft Graph authentication and the configured output endpoints.

<Aside type="tip">
Use `--dry-run` on first run to verify data is formatted correctly before committing to production ingestion.
</Aside>
